## 개발자의 시대가 끝나고 있을까?

요즘 AI에게 요구사항을 입력하면 몇 분 안에 화면과 API, 테스트 코드의 초안을 만들 수 있다. 예전에는 개발자가 직접 검색하고 작성해야 했던 코드도 coding agent가 repository를 탐색한 뒤 수정한다.

이런 모습을 보면 자연스럽게 질문하게 된다.

`이제 개발자는 필요 없어지는 것 아닐까?`

하지만 이 질문에 답하려면 먼저 `개발자`와 `코드를 작성하는 작업`을 구분해야 한다.

현재 자료는 AI가 소프트웨어 개발이라는 직업 전체를 제거하고 있다는 결론보다, 직업 안에 포함된 작업의 구성을 바꾸고 있다는 설명에 더 가깝다. 반복적인 코드 작성의 비중은 줄어들 수 있지만, 무엇을 만들어야 하는지 결정하고 결과를 검증하는 책임까지 사라졌다고 보기는 어렵다.

따라서 지금은 `개발자의 시대가 끝난다`보다 `코드 작성만으로 설명되는 개발자의 역할이 줄어들 수 있다`고 표현하는 편이 정확하다.

## 개발자 고용은 아직 증가 전망이다

미국 노동통계국(BLS)은 Software Developers, Quality Assurance Analysts, Testers의 고용이 2024년부터 2034년까지 15% 증가할 것으로 전망한다. 전체 직업 평균보다 빠른 성장률이다.

Software Developer만 분리하면 증가 전망은 16%이며, 약 267,700개의 일자리가 추가될 것으로 예상된다.

반면 전통적인 Computer Programmer 직군은 같은 기간 6% 감소할 전망이다.

BLS는 Computer Programmer의 주요 업무를 소프트웨어가 동작하도록 코드와 스크립트를 작성하고 수정하며 테스트하는 것으로 설명한다. Software Developer의 업무에는 사용자 요구 분석, 시스템 설계, 구성 요소의 결합 계획, 유지보수까지 포함한다.

두 직군의 경계가 현실에서 항상 명확한 것은 아니고, 전망 차이의 원인을 모두 AI로 설명할 수도 없다. 다만 구현 명세를 코드로 옮기는 업무와 소프트웨어 전체를 설계하고 운영하는 업무가 같은 속도로 변하지 않을 가능성을 보여주는 참고 자료는 된다.

World Economic Forum의 Future of Jobs Report 2025에서도 Software and Applications Developers는 2030년까지 빠르게 성장할 직군 중 하나로 제시된다.

동시에 조사 대상 고용주의 41%는 AI가 일부 작업을 자동화함에 따라 인력을 줄일 계획이라고 답했다.

두 전망은 모순되지 않는다.

- 소프트웨어가 필요한 산업은 계속 늘어날 수 있다.
- 일부 기업은 AI 도입과 함께 필요한 인원 구성을 바꿀 수 있다.
- 반복 작업 중심의 역할은 더 큰 압력을 받을 수 있다.
- AI 제품을 개발하고 기존 시스템에 적용하는 과정에서 새로운 개발 업무가 생길 수 있다.

전체 개발자 수요가 증가하더라도 모든 개발자가 같은 혜택을 받는 것은 아니다. 직군 내부에서 필요한 역량과 역할이 달라질 가능성이 있다.

## 직업보다 작업이 먼저 바뀐다

ILO와 NASK의 2025년 연구는 전 세계 고용의 약 25%가 생성형 AI에 잠재적으로 노출된 직업에 속한다고 분석했다. 이 수치는 실제 일자리 감소율이 아니라 AI가 업무에 영향을 줄 수 있는 범위를 의미한다.

연구가 제시하는 가장 가능성 높은 결과도 직업의 완전한 대체보다 업무의 변화다.

하나의 직업은 여러 작업으로 구성된다. 개발자도 하루 종일 새로운 코드를 입력하는 일만 하지 않는다.

일반적인 개발 업무에는 다음과 같은 작업이 섞여 있다.

- 사용자와 비즈니스 요구사항을 확인한다.
- 기존 코드와 시스템의 제약을 파악한다.
- 구현 방법과 변경 범위를 결정한다.
- 코드를 작성하고 테스트한다.
- 보안과 성능 문제를 검토한다.
- 배포 후 오류와 사용자 영향을 확인한다.
- 장애가 발생하면 원인을 추적하고 복구한다.

AI 코딩 도구가 현재 자주 활용되는 작업의 예는 다음과 같다.

- 반복적인 CRUD 구현
- 테스트와 문서 초안 작성
- 타입과 schema 생성
- 단순한 코드 변환
- 보일러플레이트 작성
- 작은 UI와 프로토타입 생성
- 범위가 명확한 버그 수정

반면 다음 작업은 조직의 맥락과 사람의 판단을 더 많이 요구할 수 있다.

- 모호한 요청을 실제 문제로 정의하기
- 여러 시스템 사이의 trade-off 결정하기
- 데이터와 권한 경계를 설계하기
- 장애 가능성과 운영 비용 판단하기
- 생성된 변경이 요구사항과 일치하는지 검증하기
- 배포 결과에 대한 책임 범위를 정하기

이 구분이 모든 프로젝트에 동일하게 적용되는 것은 아니다. 테스트가 잘 갖춰진 프로젝트에서는 agent가 넓은 범위를 처리할 수 있고, 문서화되지 않은 제약이 많은 시스템에서는 작은 변경도 사람의 확인이 많이 필요하다.

## AI는 실제로 개발자를 얼마나 빠르게 만들까?

AI 코딩 도구의 생산성 연구는 서로 다른 결과를 보여준다.

GitHub가 진행한 통제 실험에서는 95명의 전문 개발자가 같은 JavaScript HTTP server 과제를 수행했다. GitHub Copilot을 사용한 그룹은 사용하지 않은 그룹보다 평균 55% 빠르게 과제를 완료했다.

또 다른 GitHub 연구에서는 Python 경력이 5년 이상인 개발자 243명을 모집해 가상의 리뷰 웹 서버를 위한 API endpoint 구현을 요청했다. 이 가운데 202개의 유효 제출물을 분석한 결과, Copilot 그룹은 10개 단위 테스트를 모두 통과할 가능성이 53.2% 높았다. 가독성, 신뢰성, 유지보수성, 간결성 평가도 소폭 높게 나타났다.

이 결과만 보면 AI를 사용하는 개발자가 크게 유리해 보인다.

하지만 실제 코드베이스를 대상으로 한 연구에서는 다른 결과도 나왔다.

METR은 자신이 오래 기여해 온 대규모 오픈소스 프로젝트에서 작업하는 숙련 개발자 16명을 대상으로 무작위 실험을 진행했다. 연구 대상 repository는 평균 22,000개 이상의 GitHub star와 100만 줄 이상의 코드를 보유하고 있었다.

초기 2025년 AI 도구를 허용한 조건에서는 허용하지 않은 조건보다 작업 완료 시간이 19% 길었다. 참가자들은 연구 전 AI가 자신을 24% 빠르게 할 것으로 예상했고, 연구 후에도 20% 빨라졌다고 생각했지만 실제 측정 결과는 반대였다.

METR은 이 결과가 일반 개발자나 모든 개발 작업을 대표하지 않는다고 명시했다. 2026년 후속 보고에서는 최신 AI 도구가 초기 2025년보다 개발자를 더 빠르게 만들 가능성이 높다고 판단했지만, 참가자와 작업 선택의 편향, 병렬 agent 사용 시 시간 측정 문제 때문에 정확한 개선 폭을 추정하기 어렵다고 밝혔다.

이 연구들을 단순히 `AI가 빠르다` 또는 `AI가 느리다`로 정리하면 안 된다. 실험 조건이 다르기 때문이다.

실무에서 다음 조건은 AI 도입 효과를 판단할 때 확인할 만한 기준이다.

| 확인할 조건 | 살펴볼 내용 |
| --- | --- |
| 작업 범위 | 완료 조건이 명확한가? |
| 코드베이스 | 구조와 규칙이 문서화되어 있는가? |
| 검증 방법 | 결과를 자동 테스트로 확인할 수 있는가? |
| 컨텍스트 | 필요한 정보를 agent가 읽을 수 있는가? |
| 변경 위험 | 실패했을 때 영향과 복구 방법이 명확한가? |

작은 과제에서 확인된 코드 작성 속도와 실제 제품의 전체 배포 속도는 같은 지표가 아니다. 연구 결과를 자신의 조직에 적용하려면 작업 종류, 코드베이스, 검증 비용을 함께 봐야 한다.

## 더 많은 코드가 더 높은 생산성은 아니다

AI 도구는 짧은 시간에 많은 코드를 만들 수 있다. 하지만 소프트웨어 개발의 목적은 코드 양을 늘리는 것이 아니다.

생성된 코드가 많아지면 다음 작업의 범위도 커질 수 있다.

- 코드 리뷰
- 동작 테스트
- 보안 검토
- 의존성 관리
- 장애 가능성 확인
- 중복 구현 정리

모든 비용이 코드 양에 비례하는 것은 아니다. 중요한 것은 생성 속도가 빨라졌을 때 조직의 검증과 배포 과정도 그 속도를 감당할 수 있는가이다.

DORA의 State of AI-assisted Software Development 2025는 AI를 조직의 기존 강점과 약점을 확대하는 `amplifier`로 설명한다. AI 투자의 효과는 도구 자체만이 아니라 조직의 기반 시스템에 대한 투자에 달려 있다는 것이다.

이 관점에서 보면 AI 코딩 도구를 도입할 때 다음과 같은 환경을 함께 확인할 필요가 있다.

- 실행 가능한 테스트가 있다.
- 작은 단위로 변경하고 리뷰한다.
- coding convention과 architecture rule이 문서화되어 있다.
- production 상태를 확인할 수 있는 monitoring이 있다.
- 실패한 배포를 복구할 방법이 있다.
- agent가 접근할 수 있는 정보와 권한이 관리되고 있다.

AI가 빠르게 코드를 생성하더라도 검증과 배포 과정이 병목으로 남아 있다면 전체 개발 흐름에서 얻는 효과는 제한될 수 있다.

## 개발자들도 AI의 결과를 완전히 신뢰하지 않는다

Stack Overflow Developer Survey 2025에서는 응답자의 84%가 AI 도구를 사용하거나 사용할 계획이라고 답했다.

하지만 AI 출력의 정확성을 신뢰한다는 응답은 약 33%였고, 불신한다는 응답은 약 46%였다.

개발자들이 보고한 대표적인 문제도 확인할 필요가 있다.

- 66%는 `거의 맞지만 완전히 맞지는 않는 답`을 불만으로 꼽았다.
- 45.2%는 AI가 생성한 코드를 디버깅하는 데 더 많은 시간이 든다고 답했다.
- 75.8%는 향후 3~5년간 배포와 모니터링 작업에 AI를 사용할 계획이 없다고 답했다.
- 69.2%는 향후 3~5년간 프로젝트 계획에 AI를 사용할 계획이 없다고 답했다.

AI는 그럴듯한 코드를 만드는 데 유용하지만, 그 코드가 현재 서비스의 요구사항과 정확히 일치하는지는 별도로 확인해야 한다.

예를 들어 AI가 인증 API를 빠르게 구현하더라도 다음 질문은 남는다.

- 현재 권한 모델과 일치하는가?
- 개인정보가 log에 기록되지 않는가?
- 기존 session과 호환되는가?
- 실패했을 때 사용자는 어떤 상태를 보게 되는가?
- 여러 요청이 동시에 들어와도 안전한가?
- 문제가 생기면 이전 버전으로 되돌릴 수 있는가?

이 질문에 답하지 않은 코드는 production에 반영할 준비가 끝났다고 보기 어렵다.

## 주니어 개발자는 더 불리해질까?

AI가 반복적인 구현 비용을 낮추면서 주니어 개발자가 자주 맡아온 업무에도 영향을 줄 가능성이 있다.

예전에는 작은 UI 수정, CRUD API, 테스트 코드 작성 같은 작업을 통해 코드베이스의 구조와 팀의 규칙을 배울 수 있었다. 이런 작업의 일부를 AI가 처리하면 조직이 주니어에게 맡기는 업무와 평가 기준도 달라질 수 있다.

다만 현재 자료만으로 AI가 주니어 채용을 얼마나 줄였는지, 주니어 개발자가 사라질 것인지 단정할 수는 없다. 이는 별도의 채용 및 경력 추적 데이터가 필요한 문제다.

대신 개발을 시작하는 단계에서 다음 능력이 더 일찍 요구될 가능성은 있다.

- AI가 만든 코드를 그대로 제출하지 않는다.
- 코드가 동작하는 이유를 설명할 수 있다.
- 테스트로 결과를 검증할 수 있다.
- 기존 구현을 찾아 재사용할 수 있다.
- 작은 기능을 배포하고 운영 상태를 확인할 수 있다.
- 모르는 부분과 AI가 추측한 부분을 구분할 수 있다.

문법을 외우는 능력뿐 아니라 코드를 읽고 검증하는 능력도 중요한 학습 목표가 된다.

## 앞으로 개발자에게 중요한 능력

AI 시대에도 기본적인 프로그래밍 지식은 필요하다. 생성된 코드가 잘못됐는지 판단하려면 언어, 네트워크, 데이터베이스, 운영체제, 보안에 대한 이해가 도움이 된다.

그 위에서 다음 역량을 함께 준비할 필요가 있다.

### 문제 정의

`페이지를 만들어줘`보다 사용자가 어떤 문제를 해결해야 하는지 설명할 수 있어야 한다. 잘못 정의한 문제를 AI가 빠르게 구현하면 잘못된 결과를 더 빨리 얻게 된다.

### 시스템 이해

하나의 함수가 아니라 데이터 흐름, 권한, 외부 API, 배포 환경까지 함께 본다. 실제 서비스의 변경은 여러 구성 요소에 영향을 줄 수 있다.

### 검증

AI 출력은 초안으로 보고 테스트, type check, lint, code review, monitoring으로 확인한다. 생성 속도만 측정하지 말고 검증을 포함한 전체 완료 시간을 봐야 한다.

### 작은 변경 단위

agent에게 큰 기능을 한 번에 맡기면 사람이 검토해야 할 범위도 커진다. 변경을 작은 단위로 나누고 각 단계의 완료 조건을 명확히 한다.

### 도메인 지식

결제, 의료, 물류, 광고처럼 도메인 규칙이 복잡한 시스템에서는 조직의 정책과 예외 조건을 이해해야 한다. 공개된 일반 지식만으로는 내부 규칙을 모두 알 수 없다.

### 운영 관점

개발은 merge에서 끝나지 않는다. 배포 이후 오류율, 성능, 비용, 사용자 영향을 확인하고 문제가 생기면 복구해야 한다.

## AI를 사용하지 않는 것이 답은 아니다

AI가 완벽하지 않다는 이유만으로 사용하지 않는 것도 현실적인 대응은 아니다.

Anthropic은 Claude.ai와 Claude Code의 코딩 관련 상호작용 50만 건을 분석했다. Claude Code 대화의 79%는 자동화 유형으로, 21%는 사람과 협력하는 증강 유형으로 분류됐다.

코딩 관련 query 중 JavaScript와 TypeScript는 31%, HTML과 CSS는 28%를 차지했다. UI/UX component 개발과 Web·Mobile App 개발도 각각 코딩 대화의 12%, 8%를 차지했다.

이는 특정 AI 도구에서 웹 개발과 UI 구현이 활발히 사용되고 있다는 증거다. 다만 Claude 사용자만을 분석한 결과이므로 전체 개발자나 전체 AI 코딩 시장으로 일반화해서는 안 된다. Anthropic 역시 단순한 사용자용 앱과 UI 작업이 먼저 영향을 받을 가능성을 제시했지만 이를 확정된 시장 변화로 설명하지는 않았다.

중요한 것은 AI 사용 여부만이 아니라 사용 방식이다.

다음과 같은 흐름으로 활용할 수 있다.

1. 사람이 문제와 완료 조건을 정의한다.
1. agent가 관련 코드와 문서를 조사한다.
1. 작은 범위의 변경을 생성한다.
1. 자동 테스트와 정적 검사를 실행한다.
1. 사람이 설계와 동작을 검토한다.
1. production에서 결과를 관찰한다.
1. 반복되는 실패는 prompt뿐 아니라 문서, 테스트, 구조를 개선해 해결한다.

이 흐름에서 AI는 개발 시스템 안에서 반복 작업을 줄이는 도구로 사용할 수 있다.

## 마무리

AI의 등장으로 모든 개발자가 안전하다고 말할 수는 없다.

반복적인 구현의 비용이 낮아지면 일부 역할의 수요와 한 명의 개발자에게 기대되는 작업 범위가 달라질 수 있다. 특히 구현 범위가 작고 결과를 쉽게 검증할 수 있는 작업은 더 빠르게 변화할 가능성이 있다.

하지만 현재의 고용 전망과 업무 변환 연구만으로 개발자라는 직업 전체가 사라지고 있다고 결론 내릴 수도 없다. 확인할 수 있는 것은 코드 작성과 소프트웨어 개발의 관계가 바뀌고 있다는 점이다.

AI는 코드를 만들 수 있다. 개발자는 그 변경이 해결해야 할 문제와 맞는지 확인하고, 기존 시스템에 연결하며, 안전하게 배포할 수 있는 상태인지 검증해야 한다.

앞으로 중요한 질문은 `AI가 개발자를 대체할까?`에만 머물지 않는다.

`AI가 코드를 작성하는 환경에서 개발자는 어떤 판단과 책임을 맡을 것인가?`

현재 자료가 보여주는 변화는 개발자의 즉각적인 소멸보다 개발자를 평가하는 기준의 이동에 가깝다.
